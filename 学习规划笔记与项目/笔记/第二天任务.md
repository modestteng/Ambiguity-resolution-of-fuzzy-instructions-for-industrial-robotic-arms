## 下午：代码实践（2–3 小时）——把 demo 跑通 + 做一次“模糊→更清晰”的对比实验

### 1) 环境与代码（20–40 min）

核心动作就是：clone + 安装（计划里也强调这一步）。

```
git clone https://github.com/IDEA-Research/GroundingDINO.git
cd GroundingDINO
pip install -e .
```

（上面这条安装方式来自 README 的常见流程描述。）

### 2) 下载权重（10–20 min）

按 README 常见方式建 weights 目录，然后下载 Swin-T 的权重：

```
mkdir -p weights
cd weights
wget -q https://github.com/IDEA-Research/GroundingDINO/releases/download/v0.1.0-alpha/groundingdino_swint_ogc.pth
cd ..
```

（权重链接来自官方 release 下载地址。）

### 3) 跑通最小 demo（20–40 min）

用官方 demo 脚本 `demo/inference_on_a_image.py` 跑一次（你先别改代码，先跑通）：

```
CUDA_VISIBLE_DEVICES=0 python demo/inference_on_a_image.py \
  -c groundingdino/config/GroundingDINO_SwinT_OGC.py \
  -p weights/groundingdino_swint_ogc.pth \
  -i .asset/cat_dog.jpeg \
  -o outputs/demo1 \
  -t "There is a cat and a dog in the image ." \
  [--cpu-only]
```

这条命令的参数形式（`-c/-p/-i/-o/-t`，以及可选 `--cpu-only`）来自官方仓库说明片段。

> **你今天的底线：先把这条跑通。**
>  跑不通就别急着搞工业图、别急着搞 prompt 工程——那是“二楼装修”，你地基还没浇。

### 4) Day 2 的“工业场景实验”（40–60 min）

计划里要求你做的实验其实很明确：**找一张工业场景图 + 输入 prompt + 观察检测框与置信度**。
 你按这个最小闭环做：

**实验 A（模糊 prompt）**

- 图：包含多个螺丝/零件（手机随便拍一张也行）
- Prompt：
  - 英文：`"the red screw"`
  - 中文：`"左边那个螺丝"`（如果脚本对中文支持不稳定，就先用英文）

**实验 B（更具体 prompt）**

- Prompt 改成带参照物的：
  - `"the red screw near the wrench"`
  - `"the screw on the left of the motor"`

**你要记录的 3 个东西：**

1. Top-K 框里，**第 1 名和第 2 名置信度差距**（差距小 ⇒ 模糊大）
2. 模糊 prompt vs 具体 prompt：Top-K 是否“收敛”（候选变少、更集中）
3. 哪些失败案例最常见：颜色不准/遮挡/反光/零件太小

### 5) 进阶可选：不用命令行，直接用 Python（30 min，可跳）

官方也给了 Python API 形式（`load_model/load_image/predict/annotate`），你如果想把它嵌进你后续“消歧系统”里，这一步很值。

## 晚上：复盘 + 验收（20–30 min）

### 1) 按计划的 Day1–2 验收标准自测

你要能做到：

- 讲清楚“检测（detect 类别）” vs “指代落地（referring grounding 指具体实例）”
- 对“一张图 + 模糊描述”输出 **Top-K 候选物体**
- 讲明白：**为什么 Grounding 是消歧的前置步骤**（因为它把语言的不确定性，投影成视觉候选集合）

### 2) 晚上产出物（必须交）

写一个 Day2 小结（不超过 15 行）：

- **今天跑通了什么（命令/脚本名）**
- **你那张工业图里，模糊 prompt 的 Top-K 候选是什么**
- **你觉得“下一步该问一句什么澄清问题”**（为 Day3 铺垫）
  - 例：“你说的是左边那颗，还是靠近扳手那颗？”
